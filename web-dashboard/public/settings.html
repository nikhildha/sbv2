Step-by-Step Workflow

▼

The bot runs a 1-minute heartbeat with a full 5-minute analysis cycle. Each cycle follows these steps for every coin:

1

Coin Scanning

coin_scanner.py fetches the top 50 coins by 24h volume from Binance Futures. Re-scans every 4 analysis cycles (~1 hour). This ensures we always trade the most liquid markets.

2

Data Fetching (Multi-Timeframe)

data_pipeline.py pulls OHLCV candlestick data from Binance for each coin across three timeframes: 5m (execution timing), 1h (primary analysis), 4h (macro confirmation). Each fetch pulls the last 500 candles.

3

Feature Engineering

feature_engine.py computes two types of features:

HMM Input Features

  • log_return = log(close / prev_close) — captures directional momentum
  • volatility = (high - low) / close — captures intraday price range

Technical Indicators

  • RSI (14-period) — overbought/oversold for mean reversion
  • Bollinger Bands (20, 2σ) — volatility squeeze detection
  • ATR (14-period) — used for SL/TP sizing and vol filter
4

HMM Brain — Regime Classification

hmm_brain.py runs a Gaussian Hidden Markov Model with 4 states. It trains on the first 500 candles of 1h data, then predicts the current regime.

🟢 BULLISH 🔴 BEARISH 🟡 CHOP/SIDEWAYS ⚫ CRASH/PANIC

State Assignment Logic

  • Sort all 4 HMM states by their mean log-return
  • Highest return → BULL
  • Lowest return → CRASH
  • Two middle states: disambiguated by volatility — lower vol = CHOP, higher vol = BEAR

Confidence Score

The confidence is the max posterior probability from the HMM's state prediction. A 0.90 confidence means the model is 90% sure of the current regime.

5

Multi-Timeframe Confirmation

A second HMM Brain runs on 4h candles (macro timeframe). Conflicts are filtered out:

Conflict Rules

  • 1h = BULLISH but 4h = BEARISH → SKIP (timeframes disagree)
  • 1h = BEARISH but 4h = BULLISH → SKIP
  • 4h = CRASH → SKIP regardless of 1h signal
  • Both agree → HIGH CONVICTION, proceed to trade
6

Volatility Filter

Before opening any trade, check ATR / close_price ratio:

Filter Logic

  • Below 0.5% → VOL_TOO_LOW (coin is dead, no edge, fees eat profits)
  • Above 4.0% → VOL_TOO_HIGH (too wild, SL gets spiked by wicks)
  • Between 0.5% – 4.0% → Sweet spot, proceed to trade
7

Dynamic Leverage

risk_manager.py maps confidence + regime → leverage multiplier:

Condition Leverage Rationale
CRASH regime 0× (skip) Capital preservation
CHOP regime + conf ≥ 85% 15× Mean reversion, reduced risk
BULL/BEAR + conf ≥ 95% 35× High conviction trend
BULL/BEAR + conf 91–95% 25× Moderate confidence
BULL/BEAR + conf 85–91% 15× Minimum deployment threshold
conf < 85% 0× (DO NOT DEPLOY) Below minimum — no trade
8

Trade Direction

Trend Regimes

  • BULLISH → Open LONG
  • BEARISH → Open SHORT

Chop Regime (Mean Reversion)

  • RSI > 70 → Open SHORT (overbought reversal)
  • RSI < 30 → Open LONG (oversold bounce)
  • RSI 30–70 → SKIP (no edge)
9

15m Momentum Filter

For trend trades, fetches 15-minute data and checks momentum alignment:

Filter Logic

  • Going LONG but price dropped over last 5 candles (75m) → SKIP
  • Going SHORT but price rose over last 5 candles (75m) → SKIP
  • Prevents entering against short-term momentum
10

Position Sizing & ATR-Based SL/TP

risk_manager.py calculates position size using the 2% rule, then places ATR-based stops:

Position Size Formula

  • risk_amount = balance × 2%
  • stop_distance = ATR × 1.5
  • quantity = risk_amount / stop_distance

SL/TP Placement

  • Stop Loss = entry ± (ATR × 1.5) — 1.5× ATR distance
  • Take Profit = entry ± (ATR × 3.0) — 1:2 risk-reward ratio
11

Trailing SL / TP Management

tradebook.py manages active trades and implements trailing logic:

Trailing Stop Loss

  • Activation: After price moves 1× ATR in favor of the trade
  • Trail Distance: SL follows 1× ATR behind the peak price
  • SL only tightens (moves in your favor), never loosens
  • Tracks the peak price (highest high for longs, lowest low for shorts)

Trailing Take Profit

  • Activation: When price reaches 75% of the TP distance
  • Extension: TP extends by 1.5× ATR each time
  • Max Extensions: 3 (prevents runaway targets)
  • Allows profits to run while maintaining exits
12

Trade Exit (SL/TP Only)

Trades exit ONLY via price levels — regime changes do NOT trigger exits (backtest confirmed this improves returns):

Exit Triggers

  • STOP_LOSS / TRAILING_SL — SL level hit
  • TAKE_PROFIT / TRAILING_TP — TP level hit
  • MAX_LOSS — Unrealized P&L hits -30%
  • KILL_SWITCH — Portfolio down 10%+ in 24h, all trades closed

Why No Regime-Change Exits?

Backtest analysis showed regime exits hurt returns because the HMM anticipates moves, and double fees (exit + re-entry) eat into profits. The ATR-based SL/TP system handles exits more efficiently.

How HMM Works — Interactive Simulation

What is a Hidden Markov Model?

An HMM is a statistical model where the system transitions between hidden states (market regimes) that we cannot observe directly. Instead, we observe emissions (price returns, volatility) and use them to infer the most likely hidden state sequence.

🔴 Hidden States (Regimes)

  • ● BULLISH — Uptrend, positive returns
  • ● BEARISH — Downtrend, negative returns
  • ● SIDEWAYS — Low volatility chop
  • ● CRASH — High vol, sharp drops

📊 Observable Emissions

  • Log returns (price change %)
  • Volatility (rolling std dev)
  • Volume profile changes
  • ATR (Average True Range)

🧮 How Viterbi Decoding Works (Step-by-Step)

  1. Initialize: Start with initial state probabilities (π)
  2. Observe: At each timestep, observe price return & volatility
  3. Forward pass: For each state, compute max probability of arriving there
  4. Score: P(observation | state) × max(P(previous state) × P(transition))
  5. Backtrace: Walk backwards through the path to find the most likely state sequence
  6. Output: Each candle gets a regime label (BULL/BEAR/SIDE/CRASH)
